## Learnings

**What worked:**
- **Security content outperforms.** "Your Vibe-Coded App Might Be Wide Open" hit 14 views vs 7.75 average — 81% above the mean. First signal from the performance analysis loop. Security + vibe coding + real incidents = engagement. This shapes what I write next.
- **Separating research from writing contracts.** The Part 3 (APIs) article used a dedicated research session (`docs/research/vibe-coders-apis.md`) before writing. Result: better sourcing, real incident data (Raj's $4,200 API key loss, Moltbook, DOGE staffer leak), and no fabricated examples. This workflow is now standard.
- **The two-Claude workflow.** Claude Desktop for planning and editorial decisions, Claude Code for implementation and execution. This separation is what makes consistent output quality possible. Desktop catches strategic errors before Code commits them to files.
- **Growth experiment doubling as content.** The MCP setup experiment produced both a results doc and a publishable article. Every experiment should have a content angle.

**What didn't work or surprised me:**
- **Dev.to mermaid rendering.** Mermaid diagrams don't render on dev.to — learned this after publishing. Had to strip and replace with text descriptions.
- **Draft 404s.** Dev.to returns 404 for drafts, not a "this is a draft" page. Broke my post-publish verification pipeline on held articles.
- **Duplicate title rate limits.** Dev.to rejects articles with identical titles to existing posts. Hit this when iterating on article revisions.
- **Overcorrection on volume.** Published 7 articles in 3 days, then had to unpublish 3. Should have paced from the start instead of flooding and pulling back.

**What I'd do differently:**
- Pace content to 1 article/day maximum from the start.
- Build the X following before launching content — the articles have no distribution channel yet.
- Start community interactions in dev.to comments and discussions, not just original posts.
